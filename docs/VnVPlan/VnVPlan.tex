\section{General Information}

\subsection{Summary}

The V\&V (Verification and Validation) plan described in this document applies to the VoiceBridge application. VoiceBridge is an accessibility tool that is built around speech-to-text, designed for individuals with impaired or atypical speech caused by neurological or motor speech disorders. Using a customizable model, it converts slurred or non-standard speech into accurate text and control commands on standard consumer devices. The system integrates seamlessly with commonly used technology, including web browsers, enabling open-ended, customizable control for tasks such as online shopping, browsing, and digital communication. By allowing users to operate everyday applications with their own speech patterns, the product promotes greater autonomy, accessibility, and independence.

\subsection{Objectives}

\subsection{Objectives}

The primary objective of VoiceBridge is to accurately capture and interpret the intended meaning of speech from users with impaired or atypical articulation, ensuring that spoken input is reliably translated into correct text or device actions. From a verification and validation (V\&V) perspective, this objective is met only if the system’s key qualities, including accuracy, safety, reliability, and usability, can be objectively measured and confirmed through testing.

To support this objective, the V\&V plan will define clear test procedures and measurable success criteria to validate the system’s ability to transcribe impaired speech with acceptable accuracy, correctly map recognized text to intended commands, and prevent unsafe or unintended actions through confirmation and safeguard mechanisms. Transcription accuracy will be evaluated using controlled speech datasets, while command execution will be validated through scenario-based tests under both normal and error-prone conditions.

A critical focus of validation is the safe mapping of user intent when controlling web browsers and other general-purpose software, where unintended actions could have significant consequences. Because VoiceBridge relies on non-deterministic AI models, the V\&V activities will assess whether validation layers, confirmation prompts, and fallback behaviors reduce misinterpretation and unsafe execution.

Usability and accessibility are also treated as verifiable objectives. The system must demonstrate that users can complete core tasks with minimal effort on standard consumer hardware and receive clear feedback during both successful and failed interactions. The goal of empowering users with meaningful autonomy is validated by showing that users can reliably and safely achieve intended outcomes using VoiceBridge without external assistance.

\subsection{Challenge Level}

In addition to baseline verification and validation activities, limited challenge-level evaluations may be performed to further assess system robustness. These may include performance testing under increased background noise, basic accessibility checks using assistive tools, and exploratory evaluation of safeguard behavior under ambiguous or low-confidence inputs.

These challenge-level activities are optional and intended to provide additional insight into system behavior beyond baseline validation.

\subsection{SRS Verification}

\subsubsection*{3.2.1 Requirements Validation}
Each functional and non-functional requirement will be verified for clarity, feasibility, and measurability. A formal SRS Review Checklist will be applied to 
\href{https://github.com/speech-buddies/VoiceBridge/blob/main/docs/SRS-Volere/SRS.pdf}{Sections~10--16} of the SRS document, focusing on fit criteria, ambiguity, and traceability. Automated verification will be performed for measurable criteria (e.g., transcription latency, confidence scores) using \texttt{PyTest}, while manual validation will confirm qualitative attributes such as user feedback clarity.

\subsubsection*{3.2.2 Supervisor Review}
A structured review session will be held with Dr.~Christian Brodbeck. The meeting will consist of:

\begin{enumerate}
    \item A concise summary of all functional and safety-related requirements.
    \item System and use case diagrams for visual reference.
    \item Specific discussion prompts on potentially ambiguous or high-risk requirements.
\end{enumerate}

During the meeting, we will assess correctness, feasibility, and alignment with user needs. All comments will be logged as GitHub issues and tracked under the \textit{SRS Verification} label for resolution.

\subsubsection*{3.2.3 Prototype-Based Validation}
A low-fidelity prototype of VoiceBridge’s transcription interface and feedback module will be used to validate usability-related requirements (e.g., accessibility of controls, clarity of feedback, and response timing). Test participants will perform scripted scenarios derived from core functional requirements, such as initiating live transcription, adjusting speech sensitivity, and reviewing transcript accuracy.Results will be compared against defined success metrics (e.g., \textit{task completion within 5 seconds} or \textit{90\% accuracy threshold}).

\subsubsection*{3.2.4 Continuous Verification}
To ensure ongoing alignment between the SRS and the evolving design, biweekly verification reviews will be conducted. These sessions will:
\begin{itemize}
    \item Assess the impact of requirement modifications.
    \item Re-verify modified requirements using the checklist to confirm consistency and completeness.
\end{itemize}

\section{System Tests}

This section defines the system-level tests for \emph{VoiceBridge}. It follows the course template: each heading includes connective text, and each test case includes the control/type, initial state, inputs/conditions, expected outputs/results, a brief test-case derivation (where applicable), and how the test will be performed. The functional tests map directly to SRS Functional Requirements (FR1–FR5). The nonfunctional tests map to SRS Nonfunctional Requirements (LFR, UHR, PRF, OER, MSR, SCR, CLR, CPR). A traceability table linking tests to requirements is provided at the end of this section.

\subsection{Tests for Functional Requirements}

The subsections below outline tests corresponding to the SRS Functional Requirements (FR1--FR6).
Each test has a unique ID and clear execution steps.

\subsubsection{Tests for FR1 — Accept Speech Audio via Microphone}

\textbf{Goal:} Verify live microphone capture across platforms at $\geq$16 kHz.

\begin{enumerate}

\item{test-FR1-MIC-1\\}
Control: Manual

Initial State: App installed; permissions not yet granted.

Input: Speak a short phrase (\textit{``Testing one two three''}) into the default OS microphone on a Windows laptop.

Output: App requests microphone permission; once granted, waveform/level indicator appears and raw audio frames are buffered at $\geq$16 kHz.

How test will be performed: Launch app $\rightarrow$ start capture $\rightarrow$ verify UI level meter moves; export a short recording and inspect metadata (sample rate $\geq$16 kHz).
\end{enumerate}

\begin{enumerate}
\item{test-FR1-MIC-2\\}
Control: Manual

Initial State: App freshly installed on iOS and Android.

Input: First-time launch and attempt to record.

Output: OS permission prompt shown; after approval, capture begins; denial shows friendly error and retry path.

How test will be performed: Deny once (expect error + guidance), then allow (expect capture + levels).
\end{enumerate}

\begin{enumerate}
\item{test-FR1-MIC-3\\}
Control: Manual

Initial State: Multiple audio devices present (laptop microphone + USB microphone).

Input: Unplug/plug devices while capturing.

Output: The system selects the current default audio input device. If the device disconnects, it automatically falls back to another available microphone and notifies the user. If it cannot find any, it shows an error.

How test will be performed: Start capture $\rightarrow$ unplug active microphone $\rightarrow$ observe fallback notice $\rightarrow$ confirm continued capture.
\end{enumerate}

\subsubsection{Tests for FR2 — Convert Impaired Speech to Text at $\geq$80\% Accuracy}

\paragraph{test-FR2-ASR-1 Dataset Evaluation vs Baseline}

\begin{enumerate}
\item{test-FR2-ASR-1\\}
Control: Automated (batch evaluation)

Initial State: Curated impaired-speech test set with references; chosen baseline model.

Input: Run system and baseline over full test set.

Output: System achieves $\geq$80\% WER reduction vs baseline \textbf{or} meets defined WER delta per SRS; report includes per-speaker metrics.

Test Case Derivation: Directly tests Fit Criterion’s WER reduction expectation.

How test will be performed: Evaluation script computes WER, CER; export CSV; assert threshold met.
\end{enumerate}

\paragraph{test-FR2-ASR-2 Common Command Recognition (Acceptance Threshold)}

\begin{enumerate}
\item{test-FR2-ASR-2\\}
Control: Automated (scripted playback)

Initial State: Command list of 50+ common intents; audio variants (different speakers/noise).

Input: Play each command sample to the system.

Output: System produces an acceptable transcription for each command (exact or accepted synonyms); overall acceptance threshold $\geq$80\% on the command set.

Test Case Derivation: Meets “$\geq$80\% accuracy for common commands.”

How test will be performed: Align decoded text with ground truth (string/intent match); assert $\geq$80\% pass rate.
\end{enumerate}

\subsubsection{FR3 — Display Transcription for Verification}

\paragraph{test-FR3-UI-1 Real-Time Display Latency}

\begin{enumerate}
\item{test-FR3-UI-1\\}
Control: Automated (instrumented)

Initial State: Logging timestamps at (a) audio frame arrival, (b) first token shown.

Input: 100 short utterances across platforms.

Output: Time-to-first-text $\leq$2 s in $\geq$95\% of trials.

Test Case Derivation: Direct Fit Criterion.

How test will be performed: Run scripted playback or live reads with a timer, store successful and unsuccessful trials and calculate percentage.
\end{enumerate}

\paragraph{test-FR3-UI-2 Continuous Update \& Finalization}

\begin{enumerate}
\item{test-FR3-UI-2\\}
Control: Manual

Initial State: Streaming UI enabled.

Input: Speak a 10–15 s sentence; include pauses and restarts.

Output: UI shows incremental textas it is transcribed; final text locks with a small visual cue; no flickering or text loss occurs.

Test Case Derivation: Ensures transparency/trust via visible updates (FR3).

How test will be performed: Observe display during input; record the session on video; verify that the text updates smoothly and that the finalization cue appears.
\end{enumerate}

\subsubsection{FR4 — Map Text to Arbitrary Device Commands}

\paragraph{test-FR4-MAP-1 Canonical Command Set Accuracy}

\begin{enumerate}
\item{test-FR4-MAP-1\\}
Control: Automated

Initial State: 50 predefined commands with canonical phrasings and expected command representations (API/CLI/accessibility).

Input: Exact text phrases.

Output: $\geq$90\% correct mapping to the expected command representation.

Test Case Derivation: Direct Fit Criterion.

How test will be performed: Feed text into intent mapper; compare output to expected representations for accuracy.
\end{enumerate}

\paragraph{test-FR4-MAP-2 Paraphrase \& Synonym Robustness}

\begin{enumerate}
\item{test-FR4-MAP-2\\}
Control: Automated

Initial State: Same 50 intents with 2–3 paraphrases each.

Input: Paraphrased commands (e.g., “launch mail app” for “open email”).

Output: Maintains $\geq$90\% accuracy across the combined set or recorded separately for robustness.

Test Case Derivation: Realistic language coverage (FR4 rationale).

How test will be performed: Evaluate precision/recall across the set and calculate percentage.
\end{enumerate}

\paragraph{test-FR4-MAP-3 Ambiguity \& Disambiguation}

\begin{enumerate}
\item{test-FR4-MAP-3\\}
Control: Manual

Initial State: Commands with ambiguous target (“open Amazon” vs “open Amazon.ca”).

Input: Ambiguous phrasing.

Output: System asks a brief clarifying question before mapping; no unsafe default.

Test Case Derivation: Safe mapping for independence without surprises (FR4).

How test will be performed: Provide ambiguous text; verify clarification prompt and final mapping.
\end{enumerate}

\paragraph{test-FR4-MAP-4 Out-of-Domain Commands}

\begin{enumerate}
\item{test-FR4-MAP-4\\}
Control: Manual

Initial State: Intent catalog fixed; user requests unsupported app/action.

Input: “Turn on bedroom lights” (if smart-home unsupported).

Output: Clear feedback: unsupported command + suggestion (link/alternative).

Test Case Derivation: Graceful handling of arbitrary requests (FR4).

How test will be performed: Issue out-of-domain (OOD) text; verify messaging and no execution.
\end{enumerate}

\subsubsection{FR5 — Execute Commands on the Host Device}

\paragraph{test-FR5-EXEC-1 End-to-End Execution Latency \& Reliability}

\begin{enumerate}
\item{test-FR5-EXEC-1\\}
Control: Automated (instrumented E2E)

Initial State: Known-good mappings for a 50-command set.

Input: Trigger each command (text or speech $\rightarrow$ text).

Output: Visible system action completes with an overall success rate of $\geq$90\% across the command set.

Test Case Derivation: Direct Fit Criterion.

How test will be performed: Timestamp intent and effect (e.g., window focus/app open), calculate success rate over the 50-command set.
\end{enumerate}

\paragraph{test-FR5-EXEC-2 Accessibility/API Fallback Paths}

\begin{enumerate}
\item{test-FR5-EXEC-2\\}
Control: Manual/Automated

Initial State: Primary API disabled (simulate outage).

Input: Execute commands that have both accessibility and API paths.

Output: System selects fallback path, still within target latency where feasible, or reports partial degradation.

Test Case Derivation: Ensures robust execution loop (FR5 rationale).

How test will be performed: Toggle feature flags/mocks; verify fallback invocation.
\end{enumerate}

\paragraph{test-FR5-EXEC-3 Safety \& Confirmation for Destructive Actions}

\begin{enumerate}
\item{test-FR5-EXEC-3\\}
Control: Manual

Initial State: Commands include potentially destructive actions (e.g., “delete email”).

Input: Issue destructive command.

Output: Confirmation prompt appears; execution only after explicit confirmation.

Test Case Derivation: Safe device control (FR5).

How test will be performed: Attempt action, 
• verify prompt appears and execution is blocked if canceled.
\end{enumerate}

\subsubsection{FR6 — Handle Security Software Interference}

\textbf{Goal:} Verify that the system detects when browser, operating system, or antivirus security controls block or restrict automated actions and provides clear feedback to the user, including the affected command and available fallback options.

No dedicated test cases are defined for FR6 in this plan. FR6 behavior is evaluated through scenario-based testing conducted during the FR5 execution tests (test-FR5-EXEC-1 through test-FR5-EXEC-3). Any blocking events observed during those tests are recorded and their feedback quality assessed. This approach is documented here to preserve traceability between the V\&V Plan and the V\&V Report.

\subsubsection{Usability \& Humanity}

\paragraph{test-UH-1 Core Task Ease of Use}

\paragraph{test-UH-2 Onboarding Learnability \& First-Try Success}

\begin{enumerate}

\section{Unit Test Description}

The unit tests for \emph{VoiceBridge} are designed to exercise the core modules defined in the Module Interface Specification (MIS), with a focus on the control and data-flow responsibilities of each module. Our overall philosophy is:
\begin{itemize}
  \item treat each module as a black box at its public interface (access routines) while also covering important white-box behaviours such as error paths, boundary conditions, and internal state transitions;
  \item isolate heavy or external dependencies (e.g., LLM clients, browser automation, sound hardware, ASR models) using test doubles so that unit tests remain deterministic and fast; and
  \item organise tests into small, readable cases whose names document the behaviour under test (for example, \texttt{test\_transcribe\_returns\_transcript\_text} or \texttt{test\_start\_session\_already\_running\_returns\_message}).
\end{itemize}

Concretely, the unit testing strategy follows this pattern for each module:
\begin{enumerate}
  \item At least one test for normal behaviour of each public access routine.
  \item Additional tests for invalid inputs, misconfiguration, and exceptional conditions.
  \item Tests that exercise important internal state transitions where applicable (for example, audio capture state or session lifecycle).
  \item Integration-style unit tests that verify the full happy-path pipeline for the module with all collaborators mocked.
\end{enumerate}
%
The detailed inputs and outputs for each case are documented directly in the Python test code under the \texttt{test/} directory. This section summarises the implemented tests and how they relate to the MIS modules.

\subsection{Unit Testing Scope}

The core modules that have been implemented to satisfy the application's functional requirements will be the focus of unit testing. That includes the following modules:
\begin{itemize}
    \item (M4) Speech-to-Text Engine
    \item (M5) Command Orchestrator
    \item (M7) Browser Orchestrator
    \item (M8) Session Manager
    \item (M15) Audio Capture Module
    \item (M20) Feedback Module
\end{itemize}

Note that though the modules in the presentation layer are core to the application functionality, as they supply the user interface, they will be excluded. Given the simplicity of our UI, we have chosen to forgo unit tests in favor of end-to-end (E2E) and backend integration tests. With minimal frontend logic and no complex state management, unit tests would largely just verify that components render, offering little confidence in actual user outcomes. E2E tests give us far greater coverage per test by validating complete user flows, and backend integration tests ensure the data layer behaves correctly.

The remaining modules not mentioned in the above list are considered out of scope for the current MVP phase of the project; therefore, we will not populate unit test descriptions for them.

\subsubsection{Module M4: Speech-to-Text Engine}

The Speech-to-Text Engine is implemented in \texttt{app.speech\_to\_text\_engine} and is verified by \texttt{test/test\_app/test\_stt.py}. The tests cover construction, audio format validation, model readiness, conversion of raw bytes to audio streams, the asynchronous transcription pipeline, buffering of streaming audio, and optional personalization.

\begin{enumerate}
\item{\texttt{TestInit} and \texttt{TestValidateModelReady}\\}
Type: Automatic

Initial State: A \texttt{SpeechToTextEngine} instance constructed with a fully mocked \texttt{WhisperLoraAsrModel}.

Input: Different constructor arguments (with/without noise filter and personalization store) and model readiness configurations (valid, invalid, or missing model).

Output: The engine stores configuration correctly, initialises an internal model when the checkpoint is valid, and raises \texttt{InitializationError} or \texttt{ModelValidationError} when initialisation or validation fails.

Test Case Derivation: These cases directly enforce the MIS contracts for engine construction and readiness checks, ensuring that the ASR pipeline cannot be used when its dependencies are misconfigured.

How test will be performed: Instantiate the engine under different conditions and assert internal attributes and raised exceptions using the tests in the \texttt{TestInit} and \texttt{TestValidateModelReady} classes.

\item{\texttt{TestValidateAudioFormat} and \texttt{TestBytesToAudioStream}\\}
Type: Automatic

Initial State: A \texttt{SpeechToTextEngine} with default configuration.

Input: \texttt{AudioStream} instances and raw PCM byte buffers at various sample rates and amplitudes.

Output: \texttt{validateAudioFormat} correctly accepts 16~kHz audio and rejects unsupported rates; \texttt{\_bytes\_to\_audio\_stream} produces \texttt{AudioStream} objects with the expected length, sample rate, and normalised \texttt{float32} samples.

Test Case Derivation: These tests validate the preconditions for FR2 (accurate transcription) by ensuring that only well-formed audio enters the ASR model and that conversion from bytes to internal representation is correct.

How test will be performed: Call the relevant methods with synthetic audio produced in helpers such as \texttt{\_make\_audio\_bytes} and \texttt{\_make\_audio\_stream}, then assert boolean return values and array properties.

\item{\texttt{TestTranscribe}, \texttt{TestApplyPersonalization}, and \texttt{TestTranscribeIntegration}\\}
Type: Automatic

Initial State: A mocked ASR model whose \texttt{extract\_features} and \texttt{decode} methods can succeed or raise errors; optional noise filter and personalization store.

Input: Valid and invalid audio bytes, with toggled noise filtering and personalization behaviour, and injected failures in feature extraction, filtering, or decoding.

Output: On valid input, \texttt{transcribe} returns the expected transcript string; on invalid audio or model failures, it raises a runtime error with a ``Failed to transcribe'' message; when personalization is configured, profile metadata is attached to transcripts via \texttt{\_apply\_personalization}.

Test Case Derivation: These tests exercise the full unit-level ASR pipeline corresponding to FR2, including both happy path and failure modes.

How test will be performed: Use asynchronous \texttt{pytest.mark.asyncio} tests to call \texttt{transcribe}, assert return values or exceptions, and verify that collaborators (noise filter, model, personalization store) are invoked in the correct order.
\end{enumerate}

\subsubsection{Module M5: Command Orchestrator}

The Command Orchestrator is implemented in \texttt{command\_orchestrator} and verified by \texttt{test/test\_app/test\_command\_orchestrator.py}. The unit tests stub the external \texttt{google.genai} client and exercise dataclasses, custom exceptions, prompt construction, LLM interaction, response parsing, guardrails, and the high-level \texttt{process} method.

\begin{enumerate}
\item{\texttt{TestOrchestratorResponse} and \texttt{TestExceptions}\\}
Type: Automatic

Initial State: No external state; only the \texttt{OrchestratorResponse} dataclass and custom exception hierarchy.

Input: Various combinations of constructor arguments and raised exceptions.

Output: Fields such as \texttt{needs\_clarification}, \texttt{user\_prompt}, and \texttt{metadata} are stored as specified, and \texttt{InitializationError} and \texttt{InferenceError} behave as specialisations of \texttt{OrchestratorError}.

Test Case Derivation: These tests enforce the basic contracts used by higher-level orchestration logic and ensure that error handling is traceable and type-safe.

How test will be performed: Instantiate dataclasses and raise exceptions inside \texttt{pytest.raises} blocks, checking attributes and messages.

\item{\texttt{TestCommandOrchestratorInit} and \texttt{TestGetDefaultSystemPrompt}\\}
Type: Automatic

Initial State: Environment variables may or may not define \texttt{GEMINI\_API\_KEY}; prompt files may or may not exist; \texttt{google.genai.Client} may succeed or fail.

Input: Different combinations of explicit API keys, environment configuration, model IDs, and prompt file paths.

Output: The orchestrator selects the correct API key source, stores the configured model ID, loads system prompts from file when available (or falls back to a default prompt containing all required keys), and propagates initialisation failures as \texttt{InitializationError}.

Test Case Derivation: These tests ensure that the module is robust to missing configuration and that default behaviour matches the MIS design for orchestration initialisation.

How test will be performed: Use \texttt{patch.dict} and \texttt{MagicMock} to control environment variables and the \texttt{Client} constructor, then assert internal attributes and exceptions.

\item{\texttt{TestParseResponse}, \texttt{TestCallLlm}, \texttt{TestProcess}, \texttt{TestApplyGuardrails}, and \texttt{TestMultiTurnFlow}\\}
Type: Automatic

Initial State: A \texttt{CommandOrchestrator} instance with a mocked LLM client; optional conversation history.

Input: Raw LLM responses that are valid JSON, code blocks wrapped in Markdown fences, completely invalid JSON, and payloads indicating clarification or direct commands; user inputs that should be allowed or blocked by guardrails; single-turn and multi-turn conversation contexts.

Output: \texttt{\_parse\_response} returns \texttt{OrchestratorResponse} objects with correct clarification flags, prompts, commands, reasoning, and metadata; \texttt{\_call\_llm} forwards messages and system prompts with the right roles and model ID and raises \texttt{InferenceError} on API failures; \texttt{process} handles ambiguous commands via clarification, propagates conversation context, and wraps bad JSON gracefully; \texttt{apply\_guardrails} allows benign commands and blocks dangerous ones with an explanatory message.

Test Case Derivation: These tests directly support FR4 and FR5 by ensuring that textual commands are clarified, normalised, and screened for safety before execution, and that the orchestrator behaves predictably even when the LLM returns malformed responses.

How test will be performed: Use mocked \texttt{generate\_content} calls to inject specific raw response text, run asynchronous tests for \texttt{\_call\_llm} and \texttt{process}, and assert both returned objects and interactions with the LLM client.
\end{enumerate}

\subsubsection{Module M7: Browser Orchestrator}

The Browser Orchestrator module (\texttt{browser\_orchestrator}) is responsible for delegating high-level textual commands to a browser automation agent. It is verified by \texttt{test/test\_control/test\_browser\_controller.py}, which stubs the external \texttt{browser\_use} package.

\begin{enumerate}
\item{\texttt{TestRunCommand}\\}
Type: Automatic

Initial State: A stubbed \texttt{browser\_use} environment; no active browser agent.

Input: Various task strings (normal commands, empty strings) and a mocked Browser instance.

Output: The asynchronous \texttt{run\_command} function constructs a \texttt{ChatBrowserUse} instance and an \texttt{Agent} with the correct task and browser arguments, awaits \texttt{agent.run()} exactly once, returns the resulting history object, and propagates exceptions raised by the agent.

Test Case Derivation: These tests verify the MIS contract that the orchestrator translates textual tasks into browser operations without embedding browser logic itself, aligning with FR5 execution behaviour.

How test will be performed: Use \texttt{patch} to mock \texttt{ChatBrowserUse} and \texttt{Agent}, run \texttt{run\_command} under \texttt{pytest.mark.asyncio}, and assert constructor arguments, awaited calls, and returned objects.

\item{\texttt{TestEnvironmentSetup}\\}
Type: Automatic

Initial State: The module-level environment is reloaded with and without a \texttt{BROWSER\_USE\_API\_KEY}; the platform may be Windows.

Input: Environment variable dictionaries and a module reload via \texttt{importlib.reload}.

Output: The module computes \texttt{ENV\_PATH} pointing to a \texttt{.env} file two levels above, reads the API key from the environment, and (on Windows) configures the event loop policy as \texttt{WindowsProactorEventLoopPolicy}.

Test Case Derivation: These tests ensure that browser orchestration can be configured reliably across environments and platforms.

How test will be performed: Reload the module under controlled environment settings, then assert module-level constants and, on Windows, the configured event loop policy.
\end{enumerate}

\subsubsection{Module M8: Session Manager}

The Session Manager (\texttt{session\_manager}) manages the lifecycle of a long-lived browser session. It is tested in \texttt{test/test\_control/test\_session\_manager.py}, which focuses on the behaviour of \texttt{start\_session}, \texttt{stop\_session}, and \texttt{get\_browser}.

\begin{enumerate}
\item{\texttt{TestStartSession} and \texttt{TestStopSession}\\}
Type:  Automatic

Initial State: A module-level browser reference that may be \texttt{None} or an active mocked Browser instance.

Input: Calls to \texttt{start\_session()} and \texttt{stop\_session()} in different orders (first start, repeated start, stop without start, start after stop).

Output: \texttt{start\_session} creates a Browser with \texttt{keep\_alive = True}, calls \texttt{browser.start()} exactly once, sets the global browser reference, and returns either ``Browser started'' or ``Browser already running''; \texttt{stop\_session} awaits \texttt{browser.stop()}, clears the global reference, and returns ``Browser stopped'' or ``Browser already stopped'' without raising when no browser exists.

Test Case Derivation: These tests validate that the session lifecycle is robust and idempotent, matching the MIS design for shared browser resources and supporting FR5.

How test will be performed: Use \texttt{patch} to replace \texttt{Browser} with an async-mocked instance, invoke the lifecycle functions under \texttt{pytest.mark.asyncio}, and check both messages and side effects on the global browser variable.

\item{\texttt{TestGetBrowser} and \texttt{TestSessionLifecycle}\\}
Type:  Automatic

Initial State: Different configurations of the global browser reference before and after lifecycle calls.

Input: Calls to \texttt{get\_browser()} in each state and sequences of \texttt{start\_session} and \texttt{stop\_session}.

Output: \texttt{get\_browser} returns the current Browser object when running and \texttt{None} otherwise, without modifying state; a full start--stop--start cycle results in a fresh Browser instance and leaves the module in a consistent state.

Test Case Derivation: These tests ensure that code which depends on the shared browser can safely query it and that repeated lifecycle operations do not leak or reuse invalid resources.

How test will be performed: Manipulate the module-level browser reference directly in tests, call the public functions, and assert return values and instance identity.
\end{enumerate}

\subsubsection{Module M15: Real-time Audio Capture}

The Real-time Audio Capture module (\texttt{audio\_capture}) implements microphone capture, VAD-based speech detection, and buffering before transcription. It is tested in \texttt{test/test\_inputs/test\_audio\_capture.py}, which stubs out hardware and VAD dependencies.

\begin{enumerate}
\item{\texttt{TestAudioConfig}, \texttt{TestAudioState}, and \texttt{TestRealtimeAudioCaptureInit}\\}
Type:  Automatic

Initial State: No external audio devices required; configuration and state classes only.

Input: Different constructor arguments for \texttt{AudioConfig} (sample rate, chunk duration, pre-buffer and silence durations, device selection, energy thresholds) and for \texttt{RealtimeAudioCapture}.

Output: Derived fields such as \texttt{chunk\_size}, \texttt{pre\_buffer\_chunks}, and \texttt{silence\_chunks} are computed correctly; invalid configurations raise \texttt{ValueError}; initial capture state is \texttt{IDLE} with empty statistics and correctly sized pre-buffer.

Test Case Derivation: These tests confirm that configuration constraints for FR1 (audio capture) are enforced and that state machines start from a known baseline.

How test will be performed: Instantiate configuration and capture objects with different parameters and assert field values and exceptions.

\item{\texttt{TestIsSpeech}, \texttt{TestProcessAudioChunk}, \texttt{TestStartRecording}, and \texttt{TestEndRecording}\\}
Type:  Automatic

Initial State: A \texttt{RealtimeAudioCapture} instance with a mocked VAD and (optionally) an \texttt{on\_audio\_ready} callback.

Input: Synthetic audio chunks representing silence and speech, with varying energy levels, delivered while the capture state machine is in \texttt{LISTENING}, \texttt{SPEECH\_DETECTED}, or \texttt{RECORDING}.

Output: \texttt{\_is\_speech} combines VAD output and energy thresholds to classify chunks and increments error counters on VAD failures; \texttt{\_process\_audio\_chunk} manages state transitions, pre-buffering, recording buffers, silence counters, and maximum duration enforcement; \texttt{\_start\_recording} copies pre-buffered audio and resets counters; \texttt{\_end\_recording} converts buffered chunks into a normalised \texttt{float32} array, computes metadata (duration, sample rate, timestamp), updates stats, invokes callbacks once per utterance, and returns to the listening state even when callbacks raise exceptions.

Test Case Derivation: These tests correspond directly to FR1 by ensuring that the capture module correctly detects speech, segments utterances, and prepares audio for downstream processing under realistic noise and timing conditions.

How test will be performed: Drive the private methods with synthetic chunks in different states, then assert state, buffer contents, error counters, statistics, and callback invocations.

\item{\texttt{TestAudioCallback}, \texttt{TestStartStop}, \texttt{TestGetStats}, \texttt{TestListAudioDevices}, and \texttt{TestFullFlow}\\}
Type: Automatic

Initial State: A capture instance with its internal stream mocked via \texttt{sounddevice.InputStream}, and optional callbacks to capture utterances.

Input: Calls to \texttt{start()}, \texttt{stop()}, and \texttt{\_audio\_callback} under different conditions (already running, stream errors, stopped state), as well as simulated full utterances of speech followed by sufficient silence to trigger recording end.

Output: The capture object transitions between \texttt{IDLE}, \texttt{LISTENING}, \texttt{RECORDING}, and \texttt{STOPPED} as specified; stream errors are counted without crashing; stats dictionaries are returned as defensive copies; \texttt{list\_audio\_devices} filters only input-capable devices; and the full flow produces one callback per utterance with the expected audio data and metadata.

Test Case Derivation: These tests ensure that the module behaves correctly when integrated with the OS audio stack and that repeated start/stop operations and error conditions do not leave it in an inconsistent state.

How test will be performed: Use \texttt{patch} to replace \texttt{sounddevice.InputStream}, simulate callback invocations, and run integration-style flows that mimic real speech followed by silence while asserting resulting callbacks and stats.
\end{enumerate}

\subsubsection{Module M20: Feedback Module}

The Feedback Module is implemented in \texttt{feedback\_store} and is verified by \texttt{test/test\_data/test\_feedback\_module.py}. These tests focus on reliable, structured recording of user feedback about executed commands (for example, ratings and satisfaction signals) and on robustness under concurrent writes and I/O errors.

\begin{enumerate}
\item{\texttt{TestInit}\\}
Type:  Automatic

Initial State: No prior feedback log; optional explicit or default log path.

Input: Different constructor arguments for \texttt{FeedbackStore}, including explicit \texttt{log\_path} values (as strings and \texttt{Path} objects) and the no-argument default.

Output: The store always normalises \texttt{log\_path} to a \texttt{Path} object, chooses a default \texttt{feedback\_log.jsonl} file in the module directory when no path is provided, and initialises a re-entrant lock for thread-safe writes.

Test Case Derivation: These tests ensure that initialisation is predictable across environments and that the module is safe to use concurrently from different parts of the system.

How test will be performed: Instantiate \texttt{FeedbackStore} with various arguments and assert properties of \texttt{log\_path} and \texttt{\_lock}.

\item{\texttt{TestRecordStructure} and \texttt{TestSourceAutoDetection}\\}
Type:  Automatic

Initial State: A store whose log file is backed by a temporary directory.

Input: Calls to \texttt{log\_feedback} with different command IDs, feedback types, values, optional command text, and optional explicit sources.

Output: Each written JSON record contains all required keys (\texttt{uuid}, \texttt{command\_id}, \texttt{command\_text}, \texttt{type}, \texttt{value}, \texttt{source}, \texttt{timestamp}); UUIDs are valid version~4 strings and unique per record; timestamps are valid ISO-8601 values ending in \texttt{Z}; sources are automatically classified as \texttt{verbal}, \texttt{ui}, or \texttt{unknown} based on the value, unless an explicit source is provided (which is preserved as-is).

Test Case Derivation: These tests enforce the data schema used for offline analysis of user feedback and ensure that automatic source classification behaves as intended.

How test will be performed: Log feedback events, read back lines from the JSONL log, and assert field presence, formats, and classification logic.

\item{\texttt{TestFileWriting} and \texttt{TestErrorHandling}\\}
Type:  Automatic

Initial State: A store with an existing or non-existing log file path.

Input: Single and repeated calls to \texttt{log\_feedback}, with and without simulated I/O failures (for example, by patching \texttt{Path.open} to raise \texttt{IOError}).

Output: Each call appends a single valid JSON object on its own line without overwriting previous records; log files are created on demand if missing; transient write errors are swallowed so that callers are not interrupted, and such errors are reported via the module logger; subsequent calls after an error still succeed and produce valid records.

Test Case Derivation: These tests provide confidence that feedback collection is resilient to partial disk failures and that logging cannot crash the main application.

How test will be performed: Inspect the contents of the log file after different sequences of writes and use \texttt{patch} to inject controlled I/O failures while asserting logger calls.

\item{\texttt{TestThreadSafety}\\}
Type:  Automatic

Initial State: A store with an empty log file.

Input: Concurrent calls to \texttt{log\_feedback} from multiple threads.

Output: All attempted writes succeed without raising exceptions; the resulting log file contains exactly one record per attempted write, each with a unique UUID.

Test Case Derivation: These tests validate that the internal locking strategy of the Feedback Module is sufficient to prevent lost updates or corrupted records under concurrent usage.

How test will be performed: Spawn multiple threads that call \texttt{log\_feedback} in parallel, then read back all records and assert count equality and UUID uniqueness.
\end{enumerate}
