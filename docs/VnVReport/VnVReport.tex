\section{Functional Requirements Evaluation}

This section presents the evaluation results for all functional requirements as defined in the SRS. Each functional requirement is evaluated through the corresponding test cases from Section 4.1 of the V\&V Plan. For detailed test specifications (initial state, inputs, expected outputs, and execution methodology), refer to the V\&V Plan. This section focuses on test results and traceability.

\subsection{FR1: Accept Speech Audio via Microphone}

\textbf{Requirement:} The system must capture live speech input from the user through a standard built-in or external microphone, reliably detecting and recording audio from default OS microphone devices across Windows, macOS, iOS, and Android, with a minimum 16 kHz sampling rate.

\subsubsection{test-FR1-MIC-1}
\textbf{Summary:} Verify microphone permission request and audio capture.\\
\textbf{Result:} Pass -- microphone permissions were requested as expected on Windows, and audio was successfully captured at 16 kHz or higher across all test runs.

\subsubsection{test-FR1-MIC-2}
\textbf{Summary:} Verify permission handling on iOS and Android platforms.\\
\textbf{Result:} Pass -- permission handling was reliable on both iOS and Android, and microphone access was consistently verified through each platform's device permissions toolbar.

\subsubsection{test-FR1-MIC-3}
\textbf{Summary:} Verify automatic fallback when audio devices are disconnected during capture.\\
\textbf{Result:} Pass -- when the active audio device was disconnected, the system detected the change and automatically switched to the default device.

\subsection{FR2: Convert Impaired Speech to Text with $\geq$80\% Accuracy}

\textbf{Requirement:} The system must process captured audio and output textual representation. Transcription accuracy must reach at least 80\% WER reduction compared to baseline models, and achieve $\geq$80\% accuracy for common commands.

\subsubsection{test-FR2-ASR-1: Dataset Evaluation vs Baseline}
\textbf{Summary:} Automated batch evaluation comparing system WER to baseline on impaired-speech test set.\\
\textbf{Result:} Pass -- the system achieved a WER reduction greater than 80\% relative to the baseline across the impaired-speech datasets

\subsubsection{test-FR2-ASR-2: Common Command Recognition}
\textbf{Summary:} Automated evaluation of transcription accuracy for 50+ common command intents.\\
\textbf{Result:} Pass -- transcription accuracy for the 50+ common command intents exceeded 80\%, with most commands recognized correctly on the first attempt.

\subsection{FR3: Display Transcription for Verification}

\textbf{Requirement:} The transcribed text must be displayed in real time on the user's device interface. Every spoken input is displayed within 2 seconds as text on the UI, with at least 95\% consistency across trials.


\subsubsection{test-FR3-UI-1: Real-Time Display Latency}
\textbf{Summary:} Automated measurement of time-to-first-text for 100 utterances across platforms.\\
\textbf{Result:} Partial -- some utterances were displayed within approximately 2 seconds, but most trials experienced additional latency of a couple of seconds before the first text appeared.

\subsubsection{test-FR3-UI-2: Continuous Update \& Finalization}
\textbf{Summary:} Manual verification of incremental text updates and finalization cues during streaming transcription.\\
\textbf{Result:} Partial -- continuous streaming transcription is not yet implemented, but text updates do appear on the UI once recognition completes, so only finalization behavior could be partially observed.

\subsection{FR4: Map Text to Arbitrary Device Commands}

\textbf{Requirement:} The system must recognize when transcribed text corresponds to a predefined device action and translate it into the appropriate command representation.

\subsubsection{test-FR4-MAP-1: Canonical Command Set Accuracy}
\textbf{Summary:} Automated evaluation of mapping accuracy for 50 predefined commands.\\
\textbf{Result:} Pass -- commands such as ``open Chrome'', and ``play video'' were consistently mapped to the correct internal device actions with high accuracy.

\subsubsection{test-FR4-MAP-2: Paraphrase \& Synonym Robustness}
\textbf{Summary:} Automated evaluation of mapping accuracy with paraphrased and synonymous command phrasings.\\
\textbf{Result:} Pass -- paraphrased commands like ``launch Chrome'' or ``go down the page'' were correctly interpreted as their canonical equivalents (e.g., ``open Chrome'', ``scroll down'') in the vast majority of cases.

\subsubsection{test-FR4-MAP-3: Ambiguity \& Disambiguation}
\textbf{Summary:} Manual verification of clarification prompts for ambiguous commands.\\
\textbf{Result:} Pass -- when users issued ambiguous commands such as ``Try YouTube'', the system responded with clarification prompts (e.g., say yes to confirm opening YouTube) and executed the chosen option correctly.

\subsubsection{test-FR4-MAP-4: Out-of-Domain Commands}
\textbf{Summary:} Manual verification of feedback for unsupported commands.\\
\textbf{Result:} Pass -- for unsupported or out-of-domain requests like ``Hack this Website'', the system did not attempt execution and instead returned clear feedback that the command was not supported.

\subsection{FR5: Execute Commands on the Host Device}

\textbf{Requirement:} The system must execute mapped commands through the host device's accessibility framework or APIs, resulting in visible user action. 


\subsubsection{test-FR5-EXEC-1: End-to-End Execution Latency \& Reliability}
\textbf{Summary:} Automated measurement of execution latency and success rate for 50-command set across platforms.\\
\textbf{Result:} Pass -- end-to-end command execution completed successfully in roughly 5--10 seconds for most commands, with an overall success rate of about 90\%.

\subsubsection{test-FR5-EXEC-2: Accessibility/API Fallback Paths}
\textbf{Summary:} Verification of fallback behavior when primary API is disabled.\\
\textbf{Result:} Partial -- when the primary API was disabled, commands were not able to be executed and robust fallback mechanisms still need to be implemented.

\subsubsection{test-FR5-EXEC-3: Safety \& Confirmation for Destructive Actions}
\textbf{Summary:} Manual verification of confirmation prompts for destructive commands.\\
\textbf{Result:} Pass -- for destructive prompts such as deleting files or modifying the website, built-in guardrails prevented immediate execution and required explicit confirmation, effectively blocking unsafe actions.

\subsection{FR6: Handle Security Software Interference}

\textbf{Requirement:} The system must detect when browser, operating system, or antivirus security controls block or restrict automated actions and provide clear feedback to the user, including the affected command and available fallback options. No specific test cases were defined in the V\&V Plan for FR6. This requirement was evaluated through scenario-based testing during command execution tests FR5.
