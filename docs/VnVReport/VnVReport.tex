\paragraph{Calculation of Average Latency}
The average latency is calculated as:
\[
\text{Average Latency} = \frac{\sum_{i=1}^{20} L_i}{20} = 5.66 \text{ s}
\]
where \(L_i\) is the latency measured for sample \(i\).

\paragraph{Actual Result}
The measured average latency across the 20 audio samples was 5.66 seconds.

\paragraph{Conclusion}
This test \textbf{does not fully satisfy} PRF-13.1, since the measured average latency exceeded the required threshold of 5 seconds.

\subsubsection{test-PF-3: Accuracy \& Precision (Stationary Noise)}

\paragraph{Type}
Offline Evaluation

\paragraph{Test Case Derivation}
PRF-13.1, PRF-13.2

\paragraph{Test Summary}
This test was planned to evaluate ASR and command classification performance under stationary background noise conditions, including AC and fan noise mixed at 20 dB, 10 dB, and 5 dB SNR. The intended evaluation metrics were word recognition accuracy and command precision under noisy conditions.

\paragraph{Result}
This test was not executed during the current development cycle.

\paragraph{Rationale}
Robustness to stationary background noise was identified as a stretch goal for the project. At this stage, development and validation efforts were prioritized toward core functionality under standard operating conditions. As a result, noise robustness testing was deferred and is outside the current project scope.

\paragraph{Conclusion}
No result is reported for this test. Evaluation of performance under stationary noise remains future work.

\section{Unit Testing}

Unit testing for \emph{VoiceBridge} focused on the core backend modules defined in the MIS and prioritised in the V\&V Plan: the Speech-to-Text Engine (M4), Command Orchestrator (M5), Browser Orchestrator (M7), Session Manager (M8), Real-time Audio Capture (M15), and the Feedback Module (M20). For each of these modules we implemented automated tests using \texttt{pytest} that cover normal behaviour for all public access routines, representative edge cases, and key error paths (for example, invalid configuration, unavailable external services, malformed inputs, and I/O failures). Heavy external dependencies such as cloud LLM APIs, browser automation libraries, audio hardware, ASR model checkpoints, and the filesystem are replaced with test doubles where appropriate, allowing the unit tests to run deterministically and without network or device access. As a result, the unit test suite can be run quickly and repeatedly during development and in CI, and it provides direct evidence that the main control and data-flow responsibilities of each module behave as specified in the MIS.

\section{Automated Testing}
		
Automated tests are executed locally via \texttt{pytest} and in continuous integration using GitHub Actions. The CI pipeline runs the full Python unit test suite on every push and pull request, including tests under \texttt{test/test\_app}, \texttt{test/test\_control}, and \texttt{test/test\_inputs}, and fails the build if any test fails or if the test run crashes due to missing dependencies. Where necessary, platform-specific tests (for example, those that depend on Windows event loop policies) are guarded with \texttt{skip} conditions so that the suite remains portable across Linux-based CI runners and local Windows development machines. In addition to unit tests, the pipeline runs static analysis and style tools (such as \texttt{ruff} and \texttt{eslint}/\texttt{prettier} in the frontend) to catch common errors and enforce consistency before code is merged.
